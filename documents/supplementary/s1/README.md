# S1 보충자료 - 언어관리 (i18n)

## 1. i18n 키 설계 규칙
키는 `범주.페이지.스코프.기능` 구조를 따른다. 모든 키는 소문자와 점(.)으로 구분한다.
- **범주**: `app`, `nav`, `error`, `master`, `auth`, `profile`, `pet`, `log`, `booking`, `feed`
- **예시**:
  - `app.title`: 앱 전체 제목
  - `nav.home`: 하단 네비게이션 홈
  - `error.unauthorized`: 인증 오류 메시지
  - `log.blood_glucose.title`: 혈당 기록 화면 제목
  - `master.breed.pomeranian`: 견종 마스터 아이템

---

## 2. 다국어 품질 기준
- **필수 언어**: `ko` (기본), `en` (글로벌)
- **지원 언어**: `ja`, `zh_cn`, `zh_tw`, `es`, `fr`, `de`, `pt`, `vi`, `th`, `id_lang`, `ar` (총 13개국어)
- **Fallback**: 특정 언어에 값이 없을 경우 `ko` -> `en` 순으로 자동 대체한다.

---

## 3. 번역 입력 가이드 (Admin 전용)
1. **신규 키 등록**: `Admin UI > 언어관리 > 키 추가`에서 키 생성.
2. **범주/페이지 지정**: 관리를 위해 `page` 필드에 해당 화면 명시 (예: `profile_edit`).
3. **번역 작성**: `ko`, `en`을 우선 작성 후 나머지 언어는 번역기 또는 전문가 검수를 통해 채운다.
4. **활성화**: `is_active=true` 상태에서만 클라이언트로 전송된다.

---

## 4. 미번역 키 탐지 스크립트 기준
클라이언트 코드 빌드 시 또는 런타임에서 아래 동작을 수행한다.
- **UI 코드 vs DB**: `t(key)` 함수에 전달된 인자 중 DB에 존재하지 않는 키를 로그로 수집한다.
- **Empty Value Diff**: DB에 키는 존재하나 특정 언어의 값이 비어 있는 목록을 리포트로 추출한다.
